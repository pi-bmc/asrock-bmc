# X570D4I-2T Redfish Host Interface (RHI) — Session Handoff

**Goal:** make the host BIOS bring up the Redfish Host Interface over usb0 (RNDIS,
BMC = 169.254.0.17) so it authenticates to bmcweb and pushes BIOS config. Success
= **rx > 0 on the BMC's `usb0`** during/after POST (host actually transmits).

**STATUS (2026-06-14): root cause found, handler implemented + deployed, BIOS now
gets past the wall but stops at a response-format mismatch.** Details below.

---

## SESSION UPDATE 2026-06-14 (Codex) — UsbLan has no EFI_NOT_FOUND return

The decisive UEFI-shell experiment was completed over KVM while monitoring
`usb0` and dynamic-debug on the BMC:

1. The BIOS enumerates the gadget as separate `RNDIS Communications Control`
   (`22A`) and `RNDIS Ethernet Data` (`22B`) controllers.
2. `dh -p 8d8969cc-feb0-4303-b21a-1f116f385643` finds the AMI private RNDIS
   protocol on the RNDIS child/data handle. `dh -d 22B` shows it is managed by
   `AptioV USB Rndis Driver`, with no child controllers. This proves
   `UsbRndisDriverSrc.Start()` completed its pairing/install path; descriptor
   matching and ConnectController dispatch of UsbRndis are not the remaining
   problem.
3. `connect -r` and `connect 22B` do not attach an upper driver and produce no
   RNDIS control traffic.
4. The extracted matching drivers were exposed through a FAT gadget disk on
   `FS1:` and loaded successfully:
   - `load usblan.efi` -> `USB Lan Driver` handle `22D`
   - `load usbrndis.efi` -> `AptioV USB Rndis Driver` handle `22E`
5. Even after both are explicitly loaded, `connect -r` still does nothing.
   Forcing the exact binding with `connect 22B 22D` returns:

   ```
   Connect - Handle [22B] Result Not Found.
   Connect No drivers could be connected.
   ```

   This wording is the UEFI shell's aggregate `ConnectController` result; it
   does **not** prove that UsbLan itself returned `EFI_NOT_FOUND`.
6. Running `connect 22D` (try this driver against all controllers) reports
   successful connections for handles `1CE`, `1DC`, `1DE`, `203`, and `206`,
   but still does not bind `22B` or generate RNDIS traffic. The shell does not
   identify which driver ultimately won each successful connection, so this
   proves dispatch is active but does not prove UsbLan itself bound those five.
7. Throughout every test, BMC `usb0` stayed `rx_packets=0`, `tx_packets=0`, and
   dynamic-debug showed no `rndis_setup` / `rndis_msg_parser` activity.

**Complete `EFI_NOT_FOUND` audit:** there are exactly **zero** explicit
`EFI_NOT_FOUND` (`0x800000000000000e`) returns or constants in
`UsbLanDriverSrc.efi`. The four apparent `mov ..., 0xe` sites at `0x16fa`,
`0x1fb7`, `0x2080`, and `0x22a6` are 16-bit PXE/UNDI command-status values, not
EFI statuses.

The Driver Binding status paths are:

- `Supported()` at `0x488` only opens private protocol
  `8d8969cc-feb0-4303-b21a-1f116f385643` with `EFI_OPEN_PROTOCOL_BY_DRIVER`,
  closes it on success, and directly returns the Boot Services status.
- `Start()` at `0x504` directly propagates failures from opening that private
  protocol, opening DevicePath, private method `+0xc0`, and its final private
  protocol `GET_PROTOCOL`. Allocation failures explicitly return
  `EFI_UNSUPPORTED`, not `EFI_NOT_FOUND`.
- `Stop()` at `0x8b8` only propagates Boot Services statuses; it has no
  explicit `EFI_NOT_FOUND`.

The producer-side vtable installed by `UsbRndisDriverSrc` confirms that
UsbLan's early private-method calls would be externally visible:

- private method `+0xb8` -> UsbRndis `0x1010`, which issues USB I/O control
  transfers;
- private method `+0xc0` -> UsbRndis `0x1098`, which fetches/parses the MAC
  string descriptor and returns its USB I/O status.

Because no USB control activity appears, UsbLan does not reach those calls for
`22B`. The corrected failure boundary is therefore Driver Binding selection or
the initial `OpenProtocol(...8d8969cc..., BY_DRIVER)` eligibility check. Any
`EFI_NOT_FOUND` seen by the shell is synthesized/propagated outside UsbLan,
not generated by an internal UsbLan object lookup.

**Live-state caveat:** `/usr/sbin/usb_network.sh` on the running firmware is the
old image version, while configfs was hot-patched to VID/PID `046b:ffb0`.
The currently enumerated gadget still suffices to prove the above because
`AptioV USB Rndis Driver` successfully manages `22B` and installs its private
protocol.

## SESSION UPDATE 2026-06-14 (PM) — cmd-01 format SOLVED, wall moved to EFI USB-RNDIS bind

**New PROVEN facts (supersede parts of §2 below):**

1. **cmd-01 response now matches stock MegaRAC byte-for-byte** and is deployed +
   verified in-band: `[CC=0][0x52][0x01][50-byte cert-base64 fingerprint]`.
   - The `0x52` group byte is emitted via a 1-line ipmid patch
     (`0001-group-ext-use-ctx-group-for-response-echo.patch`) that makes
     `executeIpmiGroupCommand()` echo `ctx->group` instead of the raw request
     byte; the handler sets `ctx->group = 0x52`. Verified: `Context::ptr` is a
     `shared_ptr`, handler mutates the same object the framework reads back.
   - The 50-byte fingerprint is the first 50 chars of the **certificate's**
     base64 body. **BUG FIXED THIS SESSION:** bmcweb's
     `/etc/ssl/certs/https/server.pem` is a COMBINED PEM with the PRIVATE KEY
     block FIRST; the old code stripped only "CERTIFICATE" lines and so leaked
     the private-key base64 into the fingerprint. Now captures base64 strictly
     between `BEGIN/END CERTIFICATE`. In-band confirms clean `MIIC…` cert prefix.
2. **Matching cmd-01 byte-for-byte did NOT advance the BIOS.** Cold POST with the
   perfect response: BIOS still sends cmd 0x01 once (~27s after POST-complete),
   gets cc=0, then STOPS — no cmd 0x02, no `RNDIS_MSG_INIT`, `usb0` rx stays 0.
   ⇒ **The cmd-01 response format was NOT the real blocker.** §2's "current wall"
   is resolved; the wall is now the EFI USB-RNDIS bring-up (was §3 open question).
3. **EFI `UsbRndisDriverSrc.Supported()` reverse-engineered (definitive).** It
   matches the **INTERFACE** descriptor (class/subclass/protocol at desc offsets
   5/6/7) against ANY of three triples; no device-desc check, no VID/PID check.
   Accepted triples: **0x02/0x02/0xFF** (classic MS RNDIS, CDC/ACM/vendor),
   **0xEF/0x04/0x01** (RNDIS-over-IAD), **0x0A/0x00/0x00** (CDC-Data). Supported()
   is at .text RVA 0x760; comparison ladder at 0x7cc–0x7f4. (String: "Aptio V
   USB Rndis Driver".)
4. **Linux f_rndis ALREADY satisfies this at the interface level:**
   `rndis_control_intf` is HARDCODED 0x02/0x02/0xFF (matches triple #1) and the
   data interface is 0x0A/0x00/0x00 (triple #3). The configfs
   `functions/rndis.usb0/{class,subclass,protocol}` attrs only set the **IAD**
   (default 0x02/0x06/0x00) — NOT the control interface. **configfs attrs take
   DECIMAL** (e.g. `echo 255` for 0xFF; `0xff`/`0x02` mis-parse to 0).
5. **Gadget identity mismatch found:** the RUNNING firmware's
   `/usr/sbin/usb_network.sh` sets device class **0xEF/0x04/0x01** (IAD-composite);
   the REPO `usb_network.sh` was changed to **0x02/0x00/0x00** (flat CDC, stock-
   like) but that change is UNTESTED on hardware (running fw = commit 17827344a4,
   predates it). Hypothesis under test: with device class 0xEF, AMI's UsbBus may
   feed Supported() the IAD's class (0x02/0x06/0x00 → matches no triple) instead
   of the real control interface, so the driver never binds → "enumerates but
   never sends RNDIS_INIT". Device class 0x02/0x00/0x00 should force per-interface
   enumeration so Supported() sees the real 0x02/0x02/0xFF control interface.

**Fact #5 DISPROVEN by cold POST #2:** device class 0x02/0x00/0x00 fails
IDENTICALLY to 0xEF — fresh POST (08:45:12), BIOS sends cmd 0x01 again (cc=0),
but rx=0, NO `rndis_msg_parser`, same config-cycling. The gadget DOES bring up
all 3 endpoints (`RNDIS: IN/ep1 OUT/ep2 NOTIFY/ep3`), i.e. the host fully
SET_CONFIGURATIONs the device but sends ZERO RNDIS encapsulated commands — across
BOTH device-class variants. ⇒ **the descriptor/gadget is NOT the blocker; the EFI
RNDIS class driver simply never runs.**

**ROOT-CAUSE HYPOTHESIS (current lead): the UEFI Network Stack is DISABLED in the
host BIOS, so the network/RNDIS drivers are skipped entirely.** The AMI Redfish
Host Interface runs its HTTP/Redfish client over the RNDIS NIC using the UEFI
network stack (SNP/MNP/IP/TCP/HTTP); with the stack off, nothing drives RNDIS no
matter how the gadget is presented. Found in `edk2-x570d4i2t/efi-vars.json`: the
**`NetworkStackVar`** UEFI variable (Variables[7]) gates this. Required edits:

| Offset | Setting | Target |
|--------|---------|--------|
| 0 | Network Stack (UEFI net stack) | Enabled = 1 |
| 1 | IPv4 PXE Support | Disabled = 0 |
| 2 | IPv6 PXE Support | Disabled = 0 |
| 8 | Network Stack Driver Support | Enabled = 1 |

The handoff previously enabled "Network Stack Driver Support" and POST HUNG with
PXE timeouts — because PXE (offsets 1/2) was left ON. The fix is **stack ON +
PXE OFF**: the stack runs for Redfish, PXE never tries to boot. (Redfish/AMI
Redfish enable = Setup var off 55, already Enabled.)

2. **mtd6 NVRAM (persist):** host OFF → `control-bios-spi-mux-n=0` (GPIOJ1) →

**The real RHI driver chain (reverse-engineered from `edk2-x570d4i2t` PE32+ EFIs):**
- `FirmwareConfigDrv [1c5c6e7e]` is a RED HERRING — it's only the downstream AMI
  **EMA** "push BIOS attribute registry/config to Redfish" consumer; consumes AMI

**CURRENT WALL (persists through ALL descriptor/VID-PID/devclass/netstack variants):**
the host's EFI RNDIS class driver **never sends a single RNDIS control message**.
The BMC `f_rndis` gadget gets USB SET_CONFIGURATION (logs `eth_start`, carrier

**COMPLETE EFI RHI DRIVER CHAIN (all reverse-engineered) and where it breaks:**

1. `UsbRndisDriverSrc [11e32c34]` — USB driver-binding. `Supported()`@0x760 matches
   interface class {0x02/0x02/0xFF | 0xEF/0x04/0x01 | 0x0A/0x00/0x00} (our gadget
   matches). `Start()`@0x83c: when bound to the CONTROL iface it runs a pairing
   search @0x620 (find control+data interfaces), and ONLY on success allocates the

   `8d8969cc-feb0-4303-b21a-1f116f385643`** @0xbcf on a child handle. Start() itself
   sends NO RNDIS control transfer; the INITIALIZE/QUERY/SET code is in that
   protocol's method table.
2. `UsbLanDriverSrc [ca6eb4f4]` — driver-binding, **descriptor-AGNOSTIC**.
   `Supported()`@0x488 binds ANY handle exposing `8d8969cc` (OpenProtocol BY_DRIVER,

3. platform SNP/MNP/IP4 stack layers on the NII.
4. `RedfishHi [cef05965]` — gated on USB device VID/PID **0x046B/0xFFB0** (now matched).

6. `FirmwareConfigDrv [1c5c6e7e]` — EMA config push (downstream, irrelevant).

## 0. The one-paragraph summary

The BIOS performs DMTF Redfish Host Interface **credential bootstrapping over KCS
IPMI** (NetFn 0x2C group extension) *before* it will bring up usb0. Stock OpenBMC
did not implement those commands and returned `0xC1`, so the BIOS aborted RHI every
boot. We implemented them (`asrock-ipmi-oem/src/redfishhostiface.cpp`) and proved
the BIOS now invokes our handler and gets cc=0 — but it still **stops after cmd 0x01
without proceeding**, meaning it is validating and rejecting our cmd-01 *response
format*. Next: match the stock MegaRAC response exactly (`[0x52][0x01][50-byte
fingerprint]` vs our `[0x00][0x01][32-byte SHA-256]`), and/or confirm whether a
separate EFI-RNDIS-init gate is the real downstream blocker.

---

## 1. What is PROVEN (do not re-derive)

- **The host↔BMC RHI handshake is KCS IPMI `NetFn 0x2C` (Group Extension).** The
  stock MegaRAC handler is `libipmiredfishhostiface.so` (symbols `g_coreHIGroup`,
  `GetManagerCertificateFingerprint`, `GetBootstrapAccountCredentials`,
  `Generate16ByteRandomAlpNumPasswd`). Decoded via ARM reloc table:
  - `NetFn 0x2C / Cmd 0x01` = GetManagerCertificateFingerprint
  - `NetFn 0x2C / Cmd 0x02` = GetBootstrapAccountCredentials
- **EMPIRICAL (live KCS capture): the BIOS sends `NetFn 0x2C, Cmd 0x01, len=1,
  data=[0x00]`** — group/defining-body = **0x00** (NOT the DSP0270-standard 0x52),
  and NO trailing certificate-number byte. It sends this **once, near POST-complete**,
  and it is the **last** KCS command of POST.
- **Pre-fix:** our BMC returned `0xC1` → BIOS aborted (never sent cmd 0x02, rx=0).
- **Post-fix (handler registered under group 0x00):** the BIOS's `0x2C/0x01 [00]`
  now **invokes our handler** (journal: `RHI 0x2C/00/01 GetManagerCertificateFingerprint`)
  and returns **cc=0**. BUT the BIOS then **stops** (no cmd 0x02, no further KCS,
  rx=0). Same "stop after cmd01" shape as the 0xC1 case ⇒ **BIOS is rejecting our
  cmd-01 response format.**
- **All other BIOS POST IPMI is already answered correctly** (replayed all 155
  distinct commands: only `0x2C/01`, `0x06/30`, `0x0A/5C` ever returned 0xC1; the
  latter two are tolerated probes). Get-LAN ch8 returns the correct 169.254.0.17.
- **Everything else on the BMC side is correct and survives power-cycles:** u-boot
  SuperIO/iLPC2AHB gate (`SCU70`=0x7101D246 bit20=0, `HICRB`=0x0000C000 bit6=0),
  GPIOE4 hog (`devmem 0x1E780020` bit4=1), RNDIS gadget bound to vhub p1, Setup
  REDF000 (Redfish enable) = 0x01.
- **A `HostAutoFW` user already exists** on the BMC (stock AMI host-interface trusted
  user). The BIOS may auth to bmcweb as HostAutoFW rather than using cmd-02 creds.

## 2. The response-format mismatch (the current wall)

MegaRAC cmd-01 response (from ARM disasm @0x2100 of the stock lib):
`[CC=0][0x52][0x01][50-byte fingerprint]`, total len 53 (0x35). It **hard-codes
0x52** as the response group byte even though the request group is 0x00. The 50-byte
fingerprint is NOT a plain 32-byte SHA-256 — stock copies 48+2 bytes from a
`GenerateDigestMultiPartFromMem` buffer (exact encoding still TBD; reverse that fn).

Our current response: `[CC=0][0x00][0x01][32-byte SHA-256 of the bmcweb TLS cert]`.
The `0x00` comes from OpenBMC's group framework, which **echoes the request group**
and cannot be told to emit 0x52.

MegaRAC cmd-02 response (DSP0270): `[CC][0x52][Username 16B][Password 16B]`, len 34.
Ours (in-band verified working): `[CC][0x00][user16][pass16]`, and it **creates a
real BMC user** (CreateUser + PAM password) and returns the creds.

## 5. Cold POST + capture (operational) — CRITICAL CAVEATS

- **`ForceRestart` does NOT reliably re-POST this host.** Use a hard **chassis
  off→on** to guarantee a fresh POST:
  ```sh
  CH="busctl set-property xyz.openbmc_project.State.Chassis /xyz/openbmc_project/state/chassis0 xyz.openbmc_project.State.Chassis RequestedPowerTransition s"
  $CH xyz.openbmc_project.State.Chassis.Transition.Off   # wait until CurrentPowerState=Off
  $CH xyz.openbmc_project.State.Chassis.Transition.On
  ```
- The BIOS's `0x2C/01` lands **~105s into POST, right around POST-complete**.
- Watch the BIOS KCS traffic via the OEM provider's request logger:
  `journalctl -u phosphor-ipmi-host | grep "ch=3" | grep "netfn=0x2C"` (ch=3 = host
  KCS3; ch=4 = in-band busctl). The handler logs `RHI 0x2C/00/01 …` when invoked.
- Capture usb0: `tcpdump -i usb0 -n -e -s 256 -w /dev/shm/x.pcap` (BMC MAC
  02:00:16:92:54:17 = BMC-sourced; anything else = host). Watch
  `/sys/class/net/usb0/statistics/rx_packets`.
- **busctl/host-state signals are unreliable on the AST2500 vhub:** `usb0` carrier
  and the UDC `state` stay `configured`/`1` even when the host is powered OFF — do
  NOT use them to infer host enumeration. The reliable host signals are: the KCS
  `0x2C` request appearing, `RNDIS_MSG_INIT` in dmesg, and usb0 rx_packets.
- `POST Complete assert` events: `journalctl | grep "POST Complete assert"` — a NEW
  timestamp confirms a real fresh POST happened.

## 6. Recovery (host/BMC gets stuck) — AC-drain via UniFi PDU

The host can get stuck not-POSTing after repeated/aborted power cycles; only a full
**AC-drain** clears it (chassis off→on does NOT — it keeps AC applied). Outlet 13 is
governed by a UniFi **power-supervisor** that overrides simple outlet toggles.
Script: `/tmp/claude-1000/ac-drain.sh` (or rewrite). Key facts:
- Controller `https://10.0.0.1`, site `default`, API key in `X-API-KEY` header
  (keys ROTATE — ask the user for a current one; revoked key = HTTP 401 everywhere).
- **OFF/ON via power-supervisor** (the authoritative control):
  `PUT /proxy/network/v2/api/site/default/power-supervisors/6a259451d8918b39eb4dbc6c`
  body `{"power_sources":[{"client_psu_index":1,"power_source_index":13,
  "power_source_mac":"d8:b3:70:29:ae:35","power_source_type":"outlet"}],
  "settings":{"heartbeat_interval":60,"power_off_duration":120,
  "silence_threshold":900},"control_mode":"OFF"}` — `control_mode` ∈
  **{GLOBAL, MANUAL, OFF}** (NO "ON"; **GLOBAL** = restore normal power). GET on this
  v2 endpoint = 405 (PUT only).
- Legacy `outlet_overrides` PUT (`/proxy/network/api/s/default/rest/device/
  65fb7f21d4c85a182996e339`, full 20-outlet array, index 13 relay_state false/true)
  turns OFF but the supervisor prevents ON from sticking — prefer the supervisor.
- After ON: BMC returns to Redfish in **~7–8 min**; firmware gate survives the drain.
